Introduction
    What is the Linux kernel
        The Linux kernel is a program that needs to be loaded into memory and run by a boot loader, often GRUB.
        It provides an API with system calls and virtual file systems (proc, sys, debugfs) for interaction.
        The kernel enforces privileges, implements security policies, and provides controlled access to hardware and other resources.
        It is modular, allowing additional functionality to be loaded through loadable kernel modules (LKMs).
        The Linux kernel file, typically named something like vmlinuz-5.3.0-29-generic, is located in the boot directory.
        The uname -r command can be used to identify the current kernel version.
        The kernel file is compressed and loaded into memory by GRUB during the boot process.
        It's important not to delete the kernel file as it is crucial for rebooting the system.

Surveying the Linux Kernel
    Discover and control hardware
        The Linux kernel interacts with hardware by providing utilities, interfaces, and protection for hardware.
        Commands like lshw, lspci, lsusb, lsblk, and lscpu can be used to get information about hardware from the kernel.
        The hdparm command is used for getting information and setting tunables for disks.
        The kernel provides information through pseudo file systems like /proc, /dev, and /sys, which can be used to control hardware and tune kernel parameters.    
    System calls in-depth
        System calls are the interface between user space and the Linux kernel, providing services through a set of functions.
        Applications call system calls through a library, which then invokes the corresponding function in the kernel.
        Each system call has a unique number, and the kernel uses this to identify and execute the appropriate function.
        The strace utility is useful for tracing system calls made by a process, helping in debugging and understanding process behavior.
        The strace -c command can be used to summarize and count the number of system calls made by a process.
        The cd command is built into the shell and cannot be traced directly with strace; instead, you need to trace the shell itself.
        Using strace -follow and strace -O options, you can trace which process (shell or command) opens a file during redirection.
        $ strace -f -o strace.out bash date.sh
    Read messages from the kernel and /proc
        The Linux kernel prints messages using the printk function, similar to C's printf.
        These messages are stored in a RAM buffer and can be viewed using the dmesg command.
        The journalctl -k command can be used to view kernel log files.
        The proc file system (/proc) is a pseudo file system that provides up-to-date information about processes and system parameters.
        Files in /proc can be used to read system information and, in some cases, to change kernel variables.

    Introduction to /sys
        The sys file system is mounted on /sys and should be automatically mounted at boot time.
        It provides a tree structure to show how hardware is connected on your system, including buses and connections.
        The sys file system contains tens of thousands of files and directories, including valuable information for debugging and tracing kernel functions.
        It includes information about loadable kernel modules, hardware, and device information, such as block and character device files.
        strace can be used to trace system calls made by commands like lspci and fdisk.
        lspci uses the sys file system to gather information about PCI devices.
        fdisk -l attempts to open files in the sys file system to gather information about disk partitions, but it may fail to find some entries.
        The sys file system provides valuable information for debugging and tracing kernel functions, including hardware and device information.
    Device drivers and device files
        Device files are used to communicate with device drivers for block and character devices, such as keyboards and disks.
        Device files have a type (C for character, B for block) and are identified by a major and minor number, which the kernel uses to determine the appropriate driver.
        Device files are typically located in the /dev directory and are created at boot time or when the system discovers a new device.

Booting
    Understanding the bootloader GRUB
        GRUB is responsible for loading the Linux kernel into RAM and starting it.
        It also loads a small initial root file system called the initrd.
        GRUB can be interrupted before it starts the kernel, allowing for interactive use.
        It supports some file systems, enabling it to read the disk and locate files by name.
        GRUB is the most popular bootloader for Linux servers and desktops.
        Important GRUB config files are located under /etc/grub.d, and changes require running grub2-mkconfig.
        GRUB allows interactive commands and can pause before starting the kernel, which is useful for troubleshooting boot issues.
    Kernel command line parameters
        The Linux kernel supports kernel command-line parameters, which are passed by GRUB.
        These parameters can be viewed using the dmesg command or in the file /proc/cmdline.
        There are over 1,000 documented kernel parameters, which can be found in the kernel-parameters.txt file in the kernel source tree.
        Some parameters are used for debugging, security, or to work around hardware bugs, but caution is needed as incorrect parameters can cause the kernel to misbehave or not boot.
        crashkernel: Used to reserve memory for a special crash kernel to debug crashes.
            enforcing: Used to enable or disable SELinux.
            ftrace: Starts a kernel tracer early to help debug the kernel.
            init: Specifies a program to run instead of /sbin/init during the boot process.
            clearcpuid: Disables certain CPU features for security reasons.
            cachesize: Used to work around hardware bugs.
    Process 1 and start-up services
        GRUB passes control to the Linux kernel, providing command line arguments and the location of the initial RAM file system (initramfs).
        The initramfs may contain a root file system and loadable kernel modules required to mount the real root file system.
        The Linux kernel creates the first process (process ID 1), typically the init program, which is often linked to systemd.
        Systemd is responsible for starting all other processes and services, acting as the ancestor of all processes on the system.

Loadable Kernel Modules
    Loadable kernel modules (LKMs)
        The Linux kernel is modular, allowing for Loadable Kernel Modules (LKMs) to be dynamically linked into the kernel.
        LKMs are typically written in C and are compiled for specific kernel versions, making them non-binary compatible with different kernels.
        The modularity of the kernel allows for a smaller kernel file and the ability to add functionality after booting, which can reduce memory usage and enable live updates.
        LKM files are located under /lib/modules in subdirectories for each kernel version.
        These directories contain both module files and configuration files, including information about dependencies.
        Modules must match the kernel version and configuration to be loaded correctly.
        Modules run in kernel mode with full privileges, so caution is needed when writing or loading custom modules.
        LKMs are installed in the directory /lib/modules/$(uname -r) which corresponds to your kernel version.
        The directory contains subdirectories and files, including modules.dep which lists module dependencies.
        Use the command uname -r to get your kernel version and navigate to the appropriate directory.
        It's important to use a VM when working with custom modules to avoid system corruption.

    How LKMs and writing LKMs work
        Compiling a Kernel Module: To compile a kernel module, you must use the kernel makefile that corresponds to the kernel version you want to load the module into. This ensures compatibility and proper integration.
        Makefile Structure: The makefile in your directory should have a line like OBJ-M := module_name.o, where module_name is the name of your module. The kernel makefile will use this to compile your module's source code into an object file (.o) and then into a kernel object file (.ko).
        Header Files: It is crucial to use the correct kernel header files for the kernel you are building the module for. These header files contain necessary definitions and configurations that the compiler needs.
        Simple Module Structure: A basic loadable kernel module includes an initialization function and a cleanup function. The initialization function is called when the module is loaded, and the cleanup function is called when the module is removed. These functions are registered using the module_init and module_exit macros.
        Printk for Debugging: Instead of using printf, kernel modules use printk to print messages to the kernel log buffer, which can be viewed using the dmesg command. This is essential for debugging and monitoring the module's behavior.
        Module Licensing: The module's license is specified using the MODULE_LICENSE macro. Using a GPL license allows the module to access certain kernel APIs that are restricted for proprietary modules.
        Compiling a Kernel Module: Ensure you have the correct header files, make, and GCC installed. Use the kernel makefile to compile your module.
        Module Structure: A basic loadable kernel module includes an initialization function and a cleanup function, registered with module_init and module_exit macros.
        Compiler Consistency: It's crucial to use the same compiler version that was used to build the kernel to avoid misalignment issues.
        Loading and Unloading Modules: Use insmod to load your module and rmmod to remove it. Check the module's status with lsmod and view messages with dmesg.

Examining Linux Kernel Source code
    Getting Linux kernel source code
        Obtaining Kernel Source: You can download the Linux kernel source code from kernel.org using wget to get tarballs or use git clone to get the latest version from the official Torvalds Linux kernel tree.
        Using Git for Ubuntu: For Ubuntu, use Git to get the kernel source code, and determine the distribution codename from the /etc/lsb-release file.
        Shell Scripting: Utilize shell scripting to set environment variables for the distribution codename, ensuring the correct path for the Git clone command.
    Explore the kernel makefile
        Kernel Makefile Importance: The kernel Makefile is a crucial file for compiling and building the Linux kernel. It provides various targets and commands that can be executed using make.
        Make Help Command: Using make help in the top-level kernel directory provides a list of available make targets and commands. This includes commands for configuring, building, cleaning, and installing the kernel and its modules.
        Configuration File: Running a configuration command like make menuconfig or make xconfig generates a .config file in the top-level kernel directory. This text file contains all the configuration choices for the kernel build and can be shared or reused.
        Kernel Version Information: The Makefile contains variables for version information, such as the kernel version and patch level. This information is crucial for ensuring compatibility and proper identification of the kernel build.
        Build Targets: The Makefile includes various build targets like bzImage (compressed kernel image), modules (builds loadable kernel modules), modules_install (installs modules), and install (installs the kernel). Each target serves a specific purpose in the build process.
        Cleaning Targets: The Makefile also provides cleaning targets like clean, mrproper, and distclean, which are used to remove compiled files and reset the build environment.
        Kernel Configuration Interfaces: The classic interface for configuring the kernel is make menuconfig, but there are other options like make xconfig for graphical configuration.
        Building the Kernel: Running make all builds all components marked for compilation, including the kernel (vmlinux) and loadable kernel modules (bzImage). The make modules command builds all configured loadable kernel modules.
        Module Installation: The make modules_install command installs the built modules into the standard location on the system, while the make install command installs the kernel.
